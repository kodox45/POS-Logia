## Golden Zone Rules

These constraints ensure tasks are right-sized for Session 2 builders:

```
G1: Max 3 screens per frontend task
    → Split into separate tasks if a domain has 4+ screens

G2: Max 2 domains per backend task
    → Give a domain its own task if it has >8 endpoints

G3: Max 8 endpoints per backend task
    → Split CRUD endpoints from business logic endpoints if >8

G4: Every Layer 2+ task MUST have domain_packages (never empty)
    → Layer 1 tasks (infrastructure) may omit domain_packages
    → Validator tasks are EXEMPT from G4 (they validate across ALL domains)

G5: Max 8 output files per task
    → Split by component type if a domain would produce more

G6: Business rule formulas VERBATIM from domain package
    → Copy formula strings exactly — never paraphrase or summarize

G7: Cross-domain tasks include timing_notes
    → Every task involved in a cross-domain flow must reference timing
```

## Task Decomposition Algorithm

### Layer 1 — Infrastructure (no domain_package, parallel, no dependencies)

Foundation code (types, stores, routing, layouts) was already generated by T-FOUNDATION.
Layer 1 tasks handle REMAINING infrastructure that requires runtime logic.

**WRITE SCOPE RULE (CRITICAL):** Layer 1 tasks write to INFRASTRUCTURE files only
(guards, navigation components, auth components, server setup). Page content
(LoginPage.tsx, DashboardHubPage.tsx) belongs to Layer 2 domain tasks.
If a page is covered by a Layer 2 domain task, do NOT include it in Layer 1 write_scope.
This prevents two tasks writing to the same file.

```
T-001: "Implement frontend navigation and auth"
  owner: builder-frontend-core
  builder_role: "frontend"
  layer: 1, wave: 1
  goal: "Implement sidebar/tab navigation contents, auth flow UI, guard logic, shared component interiors."
  write_scope: ["src/components/navigation/", "src/components/auth/", "src/routes/guards/"]
    NOTE: Do NOT include page stubs (LoginPage, DashboardHubPage) — those belong to domain tasks
  read_sources:
    ba_files: ["architecture.json → auth_flow", "layout.json → nav_items", "roles.json"]
  business_rules: []
  state_dependencies: { fills in guard logic and nav components generated by foundation }
  acceptance: { "infra": ["Navigation renders correctly", "Auth flow works", "Guards redirect properly"] }
  depends_on: []

IF app_type != "client-only":
  T-002: "Set up backend infrastructure"
    owner: builder-backend (or builder-backend-crud if split)
    builder_role: "backend"
    layer: 1, wave: 1
    goal: "Database schema, server setup, auth middleware, base route structure."
    write_scope: ["server/", "prisma/"]
    read_sources:
      ba_files: ["architecture.json → auth_flow, data_flow", "tech-stack.json → backend"]
    business_rules: []
    acceptance: { "infra": ["DB schema created", "Server starts", "Auth middleware works"] }
    depends_on: []

IF app_type == "offline-first":
  T-003: "Implement sync engine"
    owner: builder-sync
    builder_role: "sync"
    layer: 1, wave: 1
    goal: "Sync queue implementation, conflict resolution, service worker logic."
    write_scope: ["src/services/syncService.ts", "public/sw.js"]
    business_rules: []
    acceptance: { "infra": ["Sync queue processes", "Service worker registers"] }
    depends_on: []
```

### Layer 2 — Domain Features (from domain packages)

```
FOR EACH domain_cluster (sorted by wave):

  Frontend task(s):
    screens = domain_package.screens[]
    IF screens.length <= 3:  → 1 frontend task
    IF screens.length > 3:   → split by G1 (max 3 screens per task)

    EACH task:
      owner: builder-frontend-{interface} (match interface from screen's interface field)
      builder_role: "frontend"
      layer: 2, wave: cluster.wave
      domain_packages: [".claude/implementation/domains/{cluster.domain}.json"]
      write_scope: from domain_package.write_scope (frontend paths)
      read_sources:
        primary: ".claude/implementation/domains/{cluster.domain}.json"
        entities: from domain_package.entities.owns[].name
        endpoints: { domain: cluster.domain, ids: from domain_package.endpoints[].id }
        features: from domain_package.features[].id
        screens: from domain_package.screens[].screen_id
      cross_domain_contracts: from domain_package.cross_domain_contracts[]
      business_rules: from domain_package (EXACT formulas — G6)
      state_dependencies: from domain_package.state_dependencies
      acceptance: { per feature_id: acceptance_criteria from domain_package.features[] }
      depends_on: computed by wave-based dependency injection (see below)

  WAVE-BASED DEPENDENCY INJECTION (frontend + backend):
    FOR EACH task with wave > 1:
      task.depends_on MUST include:
        1. The Layer 1 infra task of same role (always)
        2. ALL same-role tasks from wave (N-1) that share depends_on_domains[]
           (from domain_package.depends_on_domains[])
      IF domain_package.depends_on_domains[] is empty:
        Only depend on Layer 1 infra task
      This ensures wave ordering is enforced through depends_on, not just metadata

  Backend task(s) — IF app_type != "client-only":

    STEP 1 — Collect domain backend data:
      backend_domains = []
      FOR EACH domain_cluster with endpoints[].length > 0:
        backend_domains.push({ domain, endpoints, wave, features, ... })

    STEP 2 — Group by wave, then enforce G2 + G3:
      FOR EACH wave_group (domains sharing same wave):
        remaining = wave_group domains sorted by endpoint_count DESC

        WHILE remaining is not empty:
          task_domains = []
          task_endpoints = 0

          WHILE remaining is not empty
            AND task_domains.length < 2            (G2: max 2 domains)
            AND task_endpoints + next.endpoints.length <= 8  (G3: max 8 endpoints):
              task_domains.push(remaining.shift())
              task_endpoints += last_added.endpoints.length

          IF task_domains.length == 1 AND task_endpoints > 8:
            → Split this single domain: CRUD endpoints vs business logic endpoints
            → Create 2 tasks for same domain (1 CRUD, 1 logic)

          EMIT backend task from task_domains

    EACH emitted task:
      owner: builder-backend (or builder-backend-crud / builder-backend-logic if split)
      builder_role: "backend"
      layer: 2, wave: task_domains[0].wave
      domain_packages: [for each d in task_domains: ".claude/implementation/domains/{d.domain}.json"]
      write_scope: MERGE domain_package.write_scope (backend paths) from ALL task_domains
      read_sources:
        primary: domain_packages[0] (first domain)
        entities: MERGE entities.owns[].name from ALL task_domains
        endpoints: { domains: [d.domain for d in task_domains], ids: MERGE all EP-xxx IDs }
        features: MERGE features[].id from ALL task_domains
      cross_domain_contracts: MERGE from ALL task_domains
      business_rules: MERGE from ALL task_domains (EXACT formulas — G6)
      db_scope: {
        writes: MERGE write tables from ALL task_domains,
        reads: MERGE read tables from ALL task_domains
      }
      acceptance: { per feature_id: acceptance_criteria from ALL task_domains.features[] }
      depends_on: [Layer 1 backend infra task ID]

  Validation tasks (1 per validator type):
    Frontend validation:
      owner: validator-frontend
      layer: 2
      goal: "Validate all frontend features against specification."
      write_scope: ["tests/unit/", ".claude/implementation/"]
      cross_domain_contracts: all relevant validations
      depends_on: [] (Phase A starts immediately; Phase B waits for builders internally)

    Backend validation (IF app_type != "client-only"):
      owner: validator-backend
      layer: 2
      goal: "Validate all backend endpoints and business logic."
      write_scope: ["tests/api/", ".claude/implementation/"]
      depends_on: [] (Phase A starts immediately)
```

## Task Schema v5.0

Every task in the `tasks[]` array uses this structure:

```json
{
  "id": "T-001",
  "subject": "goal-oriented title",
  "owner": "teammate-name",
  "builder_role": "frontend | backend | sync | null",
  "layer": 1,
  "wave": 1,
  "description": {
    "goal": "1-2 sentence objective",
    "domain_packages": [".claude/implementation/domains/{domain}.json"],
    "write_scope": ["directories and key files this task creates"],
    "read_sources": {
      "primary": ".claude/implementation/domains/{domain}.json",
      "ba_files": [".ba/requirements/roles.json"],
      "entities": ["EntityName1", "EntityName2"],
      "endpoints": { "domains": ["domain-name"], "ids": ["EP-xxx"] },
      "features": ["F-xxx"],
      "screens": ["S-xxx"],
      "flows": ["UF-xxx"],
      "components": ["C-xxx"]
    },
    "cross_domain_contracts": [
      {
        "flow_id": "UF-xxx",
        "your_steps": [1, 2, 3],
        "context": "What happened before your steps",
        "timing_notes": ["Stock deduction at order.confirmed, NOT at payment"],
        "next_domain": "What happens after"
      }
    ],
    "business_rules": [
      {
        "rule": "Rule name",
        "type": "calculation | constraint | multi_step",
        "formula": "exact formula preserved from spec",
        "source": "F-xxx"
      }
    ],
    "state_dependencies": {
      "owns_store": "useXxxStore",
      "writes_to_tables": ["table1", "table2"],
      "reads_from_tables": ["table3"],
      "calls_other_stores": ["useYyyStore"]
    },
    "db_scope": {
      "writes": ["table1", "table2"],
      "reads": ["table3", "table4"]
    },
    "acceptance": {
      "F-xxx": ["criterion 1", "criterion 2"],
      "F-yyy": ["criterion 3"]
    }
  },
  "depends_on": ["T-xxx"],
  "on_critical_path": false,
  "active_form": "present continuous verb phrase"
}
```

### Field Rules

```
type: "calculation"  → MUST include "formula" (exact, not paraphrased)
type: "multi_step"   → MUST include "steps" array (ordered list)
type: "constraint"   → MUST include "constraint" description
cross_domain_contracts → only present for tasks in multi-domain flows
state_dependencies → for frontend tasks only (Zustand/Pinia stores)
db_scope → for backend tasks only (ORM tables). null for frontend/validator tasks
domain_packages → [] for Layer 1 tasks, NON-EMPTY for Layer 2+ domain tasks (G4)
                   Validator tasks are EXEMPT (they validate across all domains)
ba_files → additional BA source files for Layer 1 tasks (architecture.json, roles.json, etc.)
           Layer 2+ tasks read from domain_packages instead

on_critical_path → true if task has ≥3 downstream dependents (computed from depends_on graph)
                   Session 2 Lead prioritizes critical path tasks when builders are idle

acceptance format:
  Layer 2 domain tasks → MUST use feature-keyed: { "F-xxx": [...], "F-yyy": [...] }
  Layer 1 infra tasks  → MAY use generic keys: { "infra": [...] }
  Layer 3/4 tasks      → MAY use generic keys: { "sync": [...], "e2e": [...] }
  This ensures validator tasks can map acceptance criteria back to features.json
```

## Coverage Validation

Before writing the final plan, validate completeness:

```
1. Feature coverage (ALL priorities):
   FOR EACH must_have feature in features.json:
     ASSERT: feature ID appears in at least one task's read_sources.features[]
     IF missing → add to appropriate domain task or create new task (BLOCKING)
   FOR EACH should_have feature in features.json:
     CHECK: feature ID appears in at least one task's read_sources.features[]
     IF missing → LOG warning, add to closest domain task if possible (NON-BLOCKING)
   FOR EACH could_have feature in features.json:
     CHECK: feature ID appears in tasks
     IF missing → LOG info only (acceptable to omit could_have)

2. Integration-map cross-check:
   IF integration-map.json has coverage.per_feature[]:
     FOR EACH entry with status == "fully_covered" or "partially_covered":
       ASSERT: feature appears in tasks
     FOR EACH entry with status == "gap":
       LOG warning but continue (gap was pre-existing from proposal)

3. Golden Zone SELF-CHECK (run BEFORE writing file — fix violations immediately):
   FOR EACH task:
     CHECK G1: frontend tasks have ≤3 screens
       IF violated → SPLIT task by screen groups (max 3 per task)
     CHECK G2: backend tasks have ≤2 domains in domain_packages[]
       IF violated → SPLIT task by domain (re-run Step 2 grouping)
     CHECK G3: backend tasks have ≤8 endpoints
       IF violated → SPLIT into CRUD task + business logic task for same domain
     CHECK G4: Layer 2+ domain tasks have domain_packages[] non-empty
       IF violated → assign domain_packages from domain_cluster mapping
     CHECK G5: write_scope has ≤8 entries
       IF violated → SPLIT by component type (pages vs components vs services)
     CHECK G6: business_rules formulas are verbatim (spot-check 3 rules)
       IF violated → REPLACE with exact formula from domain_package
     CHECK G7: cross-domain tasks have timing_notes[]
       IF violated → ADD timing_notes from domain_package.cross_domain_contracts[]

   IMPORTANT: This is a SELF-CHECK, not just validation. If any G-rule is violated,
   FIX IT by splitting/restructuring tasks BEFORE writing the output file.
   VF3 will verify compliance, but violations should never reach VF3.

3b. Formula reconciliation:
   FOR EACH business_rule formula that appears in multiple tasks:
     VERIFY: formula text is IDENTICAL across all tasks referencing same source
     IF different wording for same calculation → use the one from the PRIMARY domain
     IF genuinely different formulas from different features → add reconciliation_note
       explaining which formula applies in which context

4. Dependency validation:
   FOR EACH task.depends_on[]:
     ASSERT: referenced task ID exists in tasks[]
     ASSERT: no circular dependencies (topological sort succeeds)

5. Owner validation:
   FOR EACH task.owner:
     ASSERT: owner appears in team_composition[].name
```
